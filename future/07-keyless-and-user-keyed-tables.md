# Keyless And User-Keyed Tables

This document proposes a major change to one of the current library assumptions:

> not every row should be forced to have a database-managed key.

Historical note:

- the selected rewrite plan in `future/plan/specification.md` keeps the spirit of this proposal,
- but refines "managed key tables" into a cleaner extension model where storage primitives stay
  agnostic and ordinary `Key` columns are synchronized by a `Keys` resource.

For many realtime workloads, row identity is not a feature. It is overhead.

Examples:

- particle simulation rows,
- transient broadphase pairs,
- contact manifolds rebuilt every frame,
- render instances generated from other state,
- audio emitters mixed from a higher-level source,
- visibility lists,
- work queues,
- temporary simulation scratch tables,
- append-only telemetry or profiling rows.

In all of those cases, maintaining:

- a generational key,
- a slot indirection,
- keyed validation logic,
- key synchronization during moves,
- event payloads that preserve stable identity,

can be wasted work.

## Core Proposal

Row identity should become a storage policy, not a universal invariant.

The future engine should support at least three identity modes:

## 1. Managed Key Tables

This is the current model.

- The engine owns generational keys.
- Rows have stable external identity through movement and compaction.
- Keyed lookup is supported.
- This is the right mode for gameplay entities and any state that needs stable references.

## 2. User-Keyed Tables

The row identity exists, but the engine does not invent it.

Examples:

- network object ID,
- asset instance ID,
- gameplay-defined handle,
- pair key like `(BodyId, BodyId)`,
- frame-local work ID generated by another system.

The table stores an indexed user-provided key column and uses that for lookup, deduplication, or replacement.

## 3. Keyless Tables

Rows have no external stable identity.

- No key column.
- No slot indirection.
- No key recycling.
- No direct `find(key)` API.
- Full freedom to reorder, compact, repartition, or rebuild rows.

This is the mode that best serves pure throughput.

## Why This Matters

## 1. Fewer Memory Touches

For keyed rows today, many operations touch:

- the row data,
- the table key array,
- the key slot array,
- and sometimes keyed validation structures.

Keyless rows only touch the row data.

## 2. Less Synchronization

Structural operations for keyed rows must preserve the identity mapping.

Keyless rows do not need:

- slot updates,
- key release/recycle,
- `keys[row] == key` validation,
- coordination for keyed event payloads.

This is especially valuable for:

- append-heavy workloads,
- rebuild-every-frame tables,
- massively parallel per-chunk generation.

## 3. More Freedom To Reorder For Locality

Once external identity disappears, the engine can reorder rows aggressively to optimize:

- chunk fullness,
- SIMD grouping,
- cache locality,
- hot/cold partitioning,
- work balancing.

This is a powerful lever that keyed storage cannot use as freely.

## 4. Simpler Fast Path

The library goal is not "everything can always be looked up by stable handle".
The goal is realtime state management at extreme performance.

Sometimes the fastest, simplest mental model is:

- this is just a bag of rows I will scan this frame.

The engine should support that directly.

## API Shape Options

## Option A: Table Builder With Identity Policy

Example:

```rust
database.table::<Projectile>().managed_keys();
database.table::<RenderInstance>().keyless();
database.table::<NetworkGhost>().user_key::<GhostId>();
```

### Good

- Explicit.
- Easy to teach.
- Makes identity a visible storage choice.

### Bad

- Pulls users toward lower-level table management APIs.

## Option B: Operation Types Encode Identity Policy

Example:

```rust
database.create::<Projectile>()?;          // managed key
database.create_keyless::<RenderInstance>()?;
database.create_indexed::<GhostId, Ghost>()?;
```

### Good

- Easier incremental adoption.

### Bad

- API proliferation.

## Option C: Schema Attributes Or Table Specs

Example:

```rust
#[table(identity = "none")]
struct RenderInstance { ... }

#[table(identity = "user", key = "GhostId")]
struct Ghost { ... }
```

### Good

- Keeps call sites clean.

### Bad

- More macro/config complexity.
- Harder when one datum should live in different table modes in different contexts.

## Recommendation

Use a table-spec or builder-level policy, not a type-level permanent policy.

Why:

- the same row shape may need different identity semantics in different subsystems,
- permanent identity baked into the datum type is too rigid.

## Query Semantics

Identity mode must affect the query API explicitly.

## Managed Key Tables

Supported:

- `Key`
- `find(key, ...)`
- `By<Key>`
- keyed joins
- identity-preserving events

## User-Keyed Tables

Supported:

- `find_by(user_key, ...)`
- joins by user key
- maybe deduplicating insert/replace semantics
- perhaps user-key event payloads instead of engine `Key`

The important question is whether the user key is:

- unique,
- indexed,
- ordered,
- hashable,
- or just a normal column.

The proposal should support at least a unique indexed mode.

## Keyless Tables

Supported:

- normal scans,
- chunk iteration,
- filter-by-table,
- joins by explicit columns or produced indices,
- per-row ephemeral index inside one query pass if needed.

Not supported:

- stable `find`,
- stable references across structural sync,
- engine `Key` events,
- APIs that imply persistent identity.

That is not a loss. It is the point.

## Events

Events need identity-aware variants.

## Managed Key Tables

Keep the current event model, perhaps improved.

## User-Keyed Tables

Emit user keys when meaningful.

Examples:

- `OnCreateUserKey<GhostId>`
- `OnDestroyUserKey<RenderBatchId>`

## Keyless Tables

Do not force per-row identity events.

Better event shapes:

- row count changed,
- chunk produced,
- table rebuilt,
- rows added/removed count,
- observer sees row contents only during local processing.

For many keyless tables, eventing may simply not be worth it.

## Structural Operations

Identity mode dramatically changes the cost model.

## Managed Key

Current expensive path:

- update slots,
- preserve key arrays,
- validate keyed lookup invariants.

## User Key

Possible strategies:

- keep a unique hash index per table,
- keep a sorted key vector for binary search,
- keep no index unless requested.

This is already more flexible than engine-managed slots because the user key can be embedded in the row layout and updated only when needed.

## Keyless

Best possible structural freedom:

- append rows,
- compact rows,
- sort rows,
- repartition chunks,
- rebuild from scratch,
- merge worker-local output buffers with no identity repair step.

This mode is especially compelling for scheduler-based frame staging:

- each thread can emit raw row batches,
- sync points merge them into keyless tables,
- no slot reconciliation exists at all.

## How This Interacts With Chunking

Chunked archetypes make this proposal much stronger.

### Managed key chunks

- keep key slices and slot repair.

### User-keyed chunks

- keep user-key index or user-key column.

### Keyless chunks

- can be almost pure column bundles.

This may justify different chunk structs internally:

- `KeyedChunk`
- `IndexedChunk<K>`
- `KeylessChunk`

or a shared chunk plus optional identity sidecar.

I prefer:

- one common chunk body for row data,
- optional identity sidecar storage selected by policy.

That keeps the hot data layout stable.

## Scheduler Synergy

This proposal is especially strong if the project adopts scheduler-first execution.

Why:

- many scheduler-produced tables are naturally keyless,
- worker threads can produce rows in parallel without coordinating identity,
- sync points can merge raw row batches cheaply,
- readonly frame queries over keyless bags are extremely efficient.

In practice, this means:

- gameplay entity world probably stays managed-key,
- ephemeral frame products become keyless by default,
- externally indexed world slices use user keys.

That is a much better match to real engine workloads than forcing one identity model everywhere.

## Risks

## Risk 1: API Fragmentation

Three identity policies can create confusing API branches.

Mitigation:

- make the identity mode visible at table creation time,
- expose only the operations that make sense for that mode,
- keep normal scan/query APIs common across modes.

## Risk 2: Query Generic Explosion

If every query must be generic over identity mode, internals get noisy.

Mitigation:

- separate identity-sensitive APIs from normal iteration APIs,
- keep most row access identity-agnostic,
- add keyed lookup traits only where required.

## Risk 3: Users Pick Wrong Mode

Some users may choose keyless where they later need stable handles.

Mitigation:

- make conversion paths exist:
  - rebuild keyless table into a table with `Key` columns plus `Keys`,
  - derive indexed view from keyless source,
  - materialize user-key index on demand.

## Recommended Adoption Order

1. Keep storage primitives key-agnostic and expose generic metadata for extension discovery.
2. Support ordinary `Key` columns plus `Keys`, and keyless tables first.
3. Make query and event APIs explicitly conditional on identity support.
4. Add user-keyed indexed tables after chunking is stable.
5. Teach the scheduler to default temporary frame products to keyless tables.

## Strong Conclusion

This proposal is high leverage.

The current stable-key design is valuable, but it should become one mode among several, not the mandatory cost every row pays.

If `that_bass` wants to maximize throughput, it should support:

- `Key`-column rows plus `Keys` for stable entities,
- user-keyed rows for externally identified data,
- keyless rows for pure bag-like, scan-heavy, rebuild-friendly state.

That change would remove a real synchronization and memory burden from some of the hottest classes of data the library is likely to manage.
